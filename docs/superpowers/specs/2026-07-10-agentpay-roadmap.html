agentpay — AI 에이전트 상거래 인프라 로드맵

관계자 논의자료 (2026-07-10) · 국제 표준의 요청측·서빙측을 모두 갖춘 하나의 코어에서, 부분집합을 골라 임베드 라이브러리 · 구축형 · 운영형 제품을 만드는 구도 · 시간축 없음(단계·순서만)
기능을 발명하지 않는다. 표준을 따른다. 각 표준에는 요청 측(에이전트·클라이언트 — 사러 가는 쪽)과 서빙 측(수용 기업·서버 — 받아주는 쪽)이 있다 — 양측을 모두 구현하면 한쪽만도 팔 수 있고, 양쪽 묶음도 팔 수 있다. 매트릭스의 칸 하나가 판매 단위다. 그리고 가드레일(보안 검문소)은 판매 이전에 우리 플랫폼 자신의 모든 요청 경로에 기본 내장된다 — 고객이 사는 것은 우리 돈 흐름 앞에 두는 것과 동일한 코드다.

용어 한 줄 — 이 자료에 반복해서 나오는 개념

DID
— 에이전트·사용자의 디지털 신분증. 위조 불가능한 서명 키에서 만들어져, 서명만 보고 "진짜 그 에이전트인지" 확인할 수 있다.
mandate (AP2)
— 사용자가 에이전트에게 써 주는 전자 위임장. "이 에이전트가 내 돈을 어디서·얼마까지·언제까지 쓸 수 있다"를 전자서명으로 못박은 문서.
VC
— 위조 여부를 기계가 검증할 수 있는 디지털 증명서 국제 표준. 졸업증명서·자격증·영수증이 모두 이 형식으로 발급될 수 있다.
가드레일 / admission
— 요청이 실행되기 전 거치는 보안 검문소 / 그 통과 판정. 이상한 요청(해킹 시도·정보 유출·규정 위반)을 거르고, 전 건을 기록한다.
custody (커스터디)
— 고객이 맡긴 돈(예치금)을 보관·기장하는 금고와 장부. 결제는 이 장부에서 차감된다.
정산 / 온체인
— 장부상 차감된 돈을 실제로 블록체인에서 송금하는 것. 블록체인 기록이라 지울 수 없는 증적이 남는다.

① 표준 × 양측 매트릭스 — 판매 단위

행 = 국제 표준, 열 = 그 표준의 두 쪽. 칸 하나하나가 따로 팔 수 있는 단위다. 점 색 = 구현 상태.

③ 집중 라인업 Ⅰ — 금융기관·핀테크: 위임증명·권한 VC 패키지

이미 실물로 돌아가는 위임 코어(위임장 발급→진위 확인→회수, 위조 시 즉시 거부) 위에, 금융권이 이미 신뢰하는 디지털 증명서(VC) 표준으로 에이전트별 권한증 발급·회수와 검증가능 영수증·감사 추적을 얹는다 — "누가·어느 에이전트에·얼마까지·언제까지"를 나중에 부인할 수 없는 서명 증거 체인으로 만든다. 회사 DID/VC 본업과 같은 표준이라 기존 금융권 신뢰 채널의 연장선이다.

유스케이스 — "졸업 증명서 발급해줘"

고객이 자기 에이전트에게 "졸업 증명서 발급해줘"라고 말하면 — 에이전트가 발급기관 카탈로그에서 증명서 발급 서비스(무형 상품)를 찾고, 사용자가 미리 서명해 둔 위임장 한도 안에서 수수료를 결제하고, 증명서가 위조 검증 가능한 VC로 발급되고, 예치금이 차감된 뒤 블록체인에서 실제 송금되어 지울 수 없는 증적이 남고, 전 과정이 VC 영수증·감사 일지로 나중에 검증 가능하다. 증명서 발급(오픈배지 = VC)은 설계에 이미 있는 상품 타입이다.

  • R1~R2 — 검문 기록부 + 권한증 관리 → "위임 + 권한증 + 정책 강제 + 감사 증적" 통합 데모, 본판매 개시
  • R3~R4 — 결제·송금·VC 영수증까지 증거 체인 완성, 요청측 SDK 포함 3배포 모델 전부

④ 집중 라인업 Ⅱ — 커머스 사업자: 에이전트 커머스 수용 패키지

AI 에이전트의 구매 트래픽을 차단할 봇이 아니라 검증된 매출로 — "누가, 어떤 권한으로" 사는지 디지털 신분증·전자 위임장으로 확인하고, 에이전트가 읽는 카탈로그 표준(ACP)으로 탐색·주문을 받고, 정산 결과는 알림(웹훅)·연동 키트로 수신한다. 내장 검문소(가드레일)가 비정상 주문을 실행 전에 거른다. 위임·결제의 서명 체인은 나중에 부인할 수 없는 증거(내부 심사·감사·소명용 — 카드사 분쟁 절차에서의 효력은 검토 항목)다.

구성도 — 운영형 기본 + 커넥터 (몰은 가볍게 연결만)

쇼핑몰 (고객사 존)

웹훅 수신 + 연동 키트기본 구매 단위 — "정산 완료" 알림 받고 주문 확정하는 연결부
ACP 카탈로그 커넥터기존 상품 DB를 에이전트가 읽는 표준 카탈로그로 변환

배포: 운영형 기본 — 금고·송금은 agentpay가 운영하고 몰은 알림·연동 키트만 받는다. 구축형 선택 시 금고 운영 책임이 고객사로 이전됨을 계약에 명시. 수요 전제(정직): 지금 판매는 에이전트 트래픽 대비 선행 구축(퍼스트무버) 제안 — 실트래픽은 쇼핑 에이전트 생태계 확산에 종속.

유스케이스 — "15만원 이하로 무선 이어폰 사줘"

소비자가 에이전트에게 "15만원 이하로 무선 이어폰 사줘"라고 말하면 — 에이전트가 몰의 카탈로그를 남은 위임 한도로 필터해 탐색하고, 몰은 들어온 요청의 신분증·위임장을 확인한 뒤 주문을 받는다. 결제는 위임장 한도 안에서만 승인·차감되고, 정산 완료가 알림으로 몰에 통지된다. 탐색 자체가 위임된 예산에 묶인다는 것이 킬러 데모다.

단계별 판매

  • 지금 — 신분증·위임장 검증 게이트 PoC (구축형/시연) + 가드레일 파생 단품 데모
  • R2B~R3 — "위임 예산 안에서만 승인·차감 + 비정상 주문 검문 차단" 라이브
  • R4 — 카탈로그 탐색·주문 + 알림·영수증 → 엔드투엔드 수용 패키지 본판매

⑤ 릴리즈 로드맵 — 공통 확정 구간 + 분기점 1개

기한 없음 — 순서와 스코프만. 🆕 신규 개발 · 🔧 기존 수정·확장 · ♻️ 그대로 재사용 · [S/M/L] 상대 크기. R1은 두 안 공통(기존 결정의 이행)이고, 실제 표결 대상은 R2 분기 하나다.

R1 신뢰 코어 — 가드레일 내장 완성 (공통 확정 구간 · 진행 중이던 작업 재개)

끝나면: 모든 요청이 검문소를 지나고 전 건이 기록부에 남는 플랫폼 — 핀테크 PoC에 "내부통제 증적" 서사 추가, 가드레일 파생 상품 데모 가능, 검문 속도 실측치 확보.
  • 🆕 검문소 조립: 즉시 판정 엔진 + AI 심층 분석의 무지연 배선 + 본체 서버 통합 + 검문 기록부M가드레일 부품(완료된 뼈대)을 실제 검문소로 조립해 플랫폼에 다는 일
  • 🔧 기존 경로(에이전트 등록·위임장 발급)에 검문소 소급 설치 — 이후 새 경로는 처음부터 검문 내장S
  • 🆕 사전 확인: 회사 기존 VC 인프라가 우리 서명 방식과 호환되는지 검증 → 표결 1의 입력S
  • ♻️ 에이전트 명부 · 위임장 관리 · 심사 규칙 (완료분 그대로)
▼ 분기점 — 표결 1: 검문소 완성 직후, 다음 한 칸을 무엇으로 채우나 ▼

R2-A 권한증(credential) 먼저 (추천 · 호환 확인 결과 조건부)

끝나면: 핀테크 본판매 개시 — "기관이 에이전트에 권한증 발급 → 제시·확인 → 회수하면 즉시 거부" + 위임 전 구간 + 기록부 증적 라이브.
  • 🆕 credential 코어 — 에이전트별 권한증(VC) 발급·확인·회수M회사의 기존 증명서 인프라를 부품 교체 방식으로 연결
  • 🆕 위임·권한증 이벤트의 감사 기록 (감사 일지 선행분)S
  • 🔧 고객 앱용 연동 키트 씨앗 — 위임장 서명 SDKS
  • 🔧 부품 분리 — 검증 게이트를 떼어 파는 라이브러리로S~M임베드 납품의 전제 조건

R2-B 금고·결제 먼저

끝나면: 돈이 실제로 도는 골격 관통 — "예치된 위임 예산 안에서만 승인·차감" 라이브, 커머스·핀테크 공통 골격.
  • 🆕 custody 코어 — 예치금 금고·장부 + 입금 감지M
  • 🆕 결제 승인 — 검문→위임장 심사→차감을 한 번에 처리하는 심장M처음부터 검문소가 배선된 상태로 만들어짐

커머스 파트너 레퍼런스 확보가 핀테크 첫 계약보다 급하다는 영업 신호가 있으면 이쪽.

R3 남은 한쪽 + 송금 개통

끝나면: 예치→승인→블록체인 송금·증적 전 구간 — AI 심층 분석이 위험 거래의 송금을 보류시키는 안전장치까지, 핀테크 풀 컴플라이언스 데모.
  • 🆕 R2에서 안 한 쪽 (권한증 또는 금고·결제)M
  • 🆕 x402 결제 절차 + 블록체인 창구(evm-gateway) — 실제 송금 실행M
  • 🆕 송금 안전장치 — 송금 전 잠깐의 보류 시간을 두고, 그 사이 도착한 AI 위험 판정만 반영·미도착 시 진행+사후 표시S
  • 🆕 외부 전문 엔진 연결 — 정책 심사(OPA)·개인정보 탐지(Presidio)M기본 모드로도 돌므로 다른 작업과 병행

R4 커머스 접점 · 완제

끝나면: 커머스 엔드투엔드 본판매(탐색→위임 한도→결제→알림·VC 영수증) + 3가지 배포 모델(임베드/구축형/운영형) 완성 — "졸업 증명서 발급" 유스케이스 풀 데모.
  • 🆕 에이전트용 카탈로그·주문 수용(ACP) + "예산 내 상품만 탐색" 킬러 데모 + 주문 검문M
  • 🆕 VC 영수증 완성 + "정산 완료" 알림(웹훅) + 어느 몰이든 붙는 표준 연동 계약M
  • 🆕 MCP 서버 · 스킬·플러그인 팩 — Claude·GPT가 agentpay를 도구로 직접 사용S
  • 🔧 로컬 AI(Ollama) 연결 · 배포 구성 정리S
  • 이후: 정책 관리 화면(G3) · 생태계 훅 어댑터(G4) · 차세대 표준 후보 확장(G5)

⑥ 전제와 열린 질문 — 답을 지어내지 않고 선점한다

예치금 보관 규제 (VASP)
운영형 = 우리가 고객 돈(스테이블코인)을 보관 → 가상자산사업자 신고 대상 여부 검토가 운영형 모델의 선행 조건. 구축형·임베드는 운영 주체가 고객사라 부담이 이전됨 — 배포 모델 선택이 곧 규제 부담 배분.
원화 결제 레일
현재 설계는 스테이블코인 송금. 카드·계좌이체는 교체 부품 자리만 파 둔 상태 — 원화 요구 시 어댑터 추가로 흡수, 스테이블코인 규제 방향은 모니터링 항목.
회사 VC 인프라 호환 (R1 사전 확인)
기존 DID/VC 인프라가 우리 서명 방식(secp256k1/EIP-712)을 지원하는지 미확인. 불일치 시 대안: 자체 발급 + 회사 인프라는 검증 쪽만 재사용, 또는 이중 서명. 표결 1은 이 확인 결과 조건부.
표준 세부 형식 적합성
AP2·A2A "done"은 기능 동작 기준. 국제 표준 문서와의 세부 형식 대조는 미실시 — 차이는 어댑터 계층에서 흡수한다는 것이 이 아키텍처의 논거.
검문 기록부의 데이터 활용 적법성
기록부를 AI 학습 자산으로 쓰려면 개인정보 제거·동의 설계 검토 필요. 감사 증적 용도와 학습 용도를 분리 설계.
가드레일 오탐·미탐 책임
납품(고객 운영) vs 운영형(우리 운영)별 보안 책임 경계를 계약으로 정의 필요. "사람 승인 대기" 보류 건의 승인 주체·제한시간 포함.
검문 속도
"속도 저하 없음"은 AI 심층 분석(비동기)에 해당. 즉시 판정은 기본 모드 기준 저지연 — R1 완료 기준에 실측 포함, 수치는 측정으로 교체. 외부 엔진 연결 시 ms대.
커머스 콜드스타트 · 경쟁 구도
에이전트 구매 실트래픽은 쇼핑 에이전트 생태계(ChatGPT·Gemini) 확산에 종속 — 지금 판매는 퍼스트무버 선행 구축 제안. 카탈로그·결제 표준은 OpenAI/Stripe 진영(ACP)과 Google 진영(AP2/x402)이 경쟁 중 — 부품 교체 구조로 양 진영을 모두 수용하는 것이 이 아키텍처의 실제 차별점.

⑦ 결정 요청 — 이 자리에서 표결할 것

  1. [표결 1 — 유일한 로드맵 분기] R1(검문소 내장) 직후의 한 칸: 권한증 관리 (R2-A, 추천 — VC 인프라 호환 확인 결과 조건부) vs 금고·결제 (R2-B). 판단 기준: 핀테크 첫 계약 vs 커머스 파트너 레퍼런스 중 어느 쪽이 급한가.
  2. [표결 2] 가드레일 "본판매 가능" 인정 기준: 기본 모드(내장 규칙)만으로 인정 vs 외부 전문 엔진(OPA·Presidio) 연결까지 필수.
  3. [표결 3] 감사 기록의 분할 출시: 위임·권한증 증적만 조기 출시(R2-A 포함) vs 결제 실거래 생긴 뒤 일괄(R3~R4).
  4. [표결 4] 고객 앱용 연동 키트(SDK) 시점: R2에서 씨앗 vs R4에서 알림·표준 연동 계약과 묶음.
  5. [표결 5] 가드레일 단독판매 출시 시점: 내장 완성(R1) 즉시 파생 출시 vs 자사 결제 트래픽 실적(R2-B/R3 이후) 확보 후 "우리가 우리 돈 앞에 두고 쓴다" 근거와 함께 출시.
