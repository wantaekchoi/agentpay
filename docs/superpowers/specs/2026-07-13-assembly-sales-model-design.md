# 조립형 판매 모델 설계 (Assembly Sales Model)

> 작성 경위: 2026-07-13 세션 — 요구 수집 → 접근안 3개(A 게이트 모델 / B 전면 재편 / C 각주 승격) 중 A 확정 → 피치 v0~v1.2 반복 → 요구사항 통합 전면 재작성.

## 1. 목적·독자

이 문서는 agentpay의 **판매·납품 방식**을 결정한다 — 무엇을 만들지(빌드 순서·로드맵)가 아니라, **만든 것을 어떤 단위로 · 누구에게 · 어떤 순서로 파는가**.

- 정본(이 문서)의 독자: 관계자(팀장·영업·개발) — 결정과 근거를 검증하는 사람.
- 파생물(피치 html)의 독자: **사전 맥락 0의 영업사원/개발팀장** — "이런 걸 만들어서 이렇게 팔면 어때요?"라는 질문을 받고 처음 읽는 사람. 콜드 리딩 가능해야 한다(30초 요약 선행, 문외한 평문·비유).

## 3. 핵심 결정

### D1. 두 층 구조 — 진열대(상품)와 부품표(BOM)  ← R2·R4

**결정:** 고객에게 보이는 앞면은 **문제 단위 상품 3개**(진열대), 부품은 견적·납품 때 꺼내는 뒷면 BOM.

- 진열대: ① **AI 검문소(가드레일) 단품** — 유일하게 단독으로 문제가 완결되는 진입 상품(land) ② **핀테크 세트**(위임·권한 증명) ③ **커머스 세트**(에이전트 손님 받기).
- identity·audit 등의 단품은 진열하지 않는다 — 어떤 고객 문제도 단독으로 해결하지 않기 때문.
- BOM의 상업적 가치 3가지: **빼기**(이미 있는 부품은 견적에서 제외 — 붙박이 일체형이 못 하는 것), **더하기**(land→expand 확장 경로), **꽂힌다**(임베드 가능 신호).

**근거:** 고객은 모듈이 아니라 문제 해결을 산다(R4). 조립형의 가치는 "잘게 판다"가 아니라 "빼고 더할 수 있다"에 있다.
**기각 대안:** 부품 전면 진열(초안 v0의 함정 — 나사·볼트를 사는 고객은 없음).

### D2. 부품 자격 5조건 게이트  ← R2·R3

**결정:** 아래 5조건을 통과한 모듈만 BOM에 "단독 납품 가능(부품)"으로 오른다. **기능 완성 ≠ 부품 자격.**

1. **순수한 알맹이** — 스프링·DB·타 부품 없이 컴파일·동작. 공유 의존은 commons(원자 부품)만.
2. **콘센트 규격** — 외부 도구·DB 연결은 전부 포트/어댑터.
3. **심어도·세워도** — 라이브러리 임베드 + 독립 서버(HTTP) 두 형태 제공.
4. **혼자서도 일한다** — 타 부품 없이 실사용 시나리오가 도는 것을 자동 시험으로 증명 (예: delegation은 우리 identity 없이 외부 DID 서명·조건 검증).
5. **설명서 동봉** — 독립 퀵스타트 + 5분 데모.

**근거:** R3("하나하나 쓸 수 있어야")를 검증 가능한 기준으로 변환. 4번이 "쪼갠 의미"의 시험이고, 3번이 D1의 "꽂힌다"를 기술적으로 보증.
**기각 대안:** 모듈이면 자동으로 부품(검증 장치 없음 — "쪼갰지만 하나하나 못 쓰는" 상태를 못 막음).
**현황(정직):** 오늘 5조건 전부 통과한 부품 0개. 가드레일이 1호 임박(①②④ 충족, ③ 서버모드 완성, ⑤ 퀵스타트 남음). identity·delegation은 기능 완성이나 `:app` 붙박이 — R2 모듈 분리가 곧 부품 자격 작업.

### D3. 전 부품 라이브러리+서버 겸용  ← R5

**결정:** 가드레일에서 검증한 모델(순수 코어 라이브러리 + 같은 부품을 담은 서버 모드)을 전 부품의 표준 제공 형태로.

**근거:** 레거시 금융권은 임베드를, 비Java 고객은 HTTP를 원한다 — 하나만 고르면 시장 절반을 버림. 추가 설계 비용이 가장 낮음(G1이 이미 증명).
**기각 대안:** 임베드 온리(비Java 배제) / 부품별 마이크로서비스 온리(임베드 수요 배제 + 고객 운영 부담).

### D4. 클라이언트 키 3단 폴백 (Signer 어댑터 사다리)  ← R6

**결정:** `agentpay-client` 부품 안에 Signer 인터페이스를 두고, 뒤에 어댑터 3종: ⓐ **회사 DID 월렛·인증 앱 연계**(1순위 — 기존 사용자 기반 재활용, R1 VC 호환 스파이크가 유효성 관문) ⓑ **SDK 임베디드 키**(자체 보유 폴백 — 고객 앱 안에서 키 생성·디바이스 보안영역 보관, 사용자는 월렛 개념 노출 0) ⓒ **BYO 외부 월렛**(크립토 네이티브용 옵션).

**근거:** 서버는 서명 검증만 하므로(개인키 비보관 불변) 어느 단이든 **서버 설계 불변** — 월렛 문제는 클라이언트 포장 문제다. ⓐ가 막혀도(월렛 팀 로드맵·앱 심사·호환 실패) ⓑ로 제품이 성립.
**기각 대안:** 회사 월렛 전제 고정(협조·호환 리스크에 제품 생사가 걸림) / 완전 커스터디 위임(위임장의 암호학적 증명 사슬 단절 + VASP 정면).

### D5. 도입 사다리 4단계 + 관찰(shadow) 모드  ← R8

**결정:** 진입장벽 4개에 각각 낮추는 장치를 제품에 내장하고, 도입을 사다리로 설계: **0단계 관찰**(차단 없이 기록만 — 무위험 PoC, 잡힌 인젝션·PII 리포트가 구매 근거) → **1단계 검문**(차단·승인 게이트 = 첫 유료) → **2단계 신원·위임** → **3단계 결제**.

| 장벽 | 장치 |
|---|---|
| 닭-달걀 ("에이전트 손님이 아직 없는데") | 검문소는 에이전트 이전에도 사내 AI 챗봇·자동화에 오늘 가치 — "지금 문제"로 판매 |
| 크립토·규제 (USDC·월렛·VASP) | 사용자 크립토 노출 0(D4-ⓑ·커스터디) + **원화 레일(기존 PG·계좌) 1급 승격 검토** — 크립토는 옵션 레일로 |
| 통합 부담 | 관찰 모드(GuardrailConfig enforcement 플래그 1개) + D3 겸용 형태 + 커넥터 방식 |
| 신뢰 (작은 팀 새 플랫폼) | 표준 포맷+무버전 API = **낮은 이탈 비용**("언제든 떠날 수 있게 만들었으니 믿으라") + 사내 DID/VC·금융권 레퍼런스 |

**기각 대안:** 정면 돌파 영업(장벽을 제품이 아니라 영업력으로 해결 — 재현 불가).

### D6. 포지셔닝 — "기본이라서 사는 인프라" + wedge 전략  ← R7

**결정:** "너무 크다"에는 **wedge**로 답한다(파는 건 플랫폼이 아니라 검문소 단품 하나, 큰 그림은 로드맵). "너무 기본이다"에는 **인프라 포지션**으로 답한다(PG·TLS처럼 만들 수 있지만 아무도 직접 만들어 유지하지 않는 자리). build-vs-buy 정면 답 3가지: ① 유지가 세금(표준 전쟁 중 wire-format 추적을 우리가 흡수) ② 틀리면 비쌈(위임·검문 증명 사슬은 감사·규제 품질이 생명) ③ 우리만 되는 연결(사내 DID/VC 인프라·금융권 신뢰 — 진짜 해자).

**기각 대안:** 플랫폼 통판매(작은 팀 신뢰 장벽에 정면충돌) / 기술 우위 주장(regex·룰 엔진은 해자가 아님 — 정직하지도 않음).

## 4. 기존 로드맵과의 접합

- **직교 관계:** 로드맵 §2 표준×양측 매트릭스 = "이 부품이 무엇을 하나", 본 모델 = "어떻게 사나". 로드맵 재작성이 아니라 **증보**.
- **R2 "모듈 분리"의 완료 기준 = D2 5조건.** 빌드 순서는 변경 없음 — "분리가 끝났다"의 정의만 명문화.
- **표결 +1 (6번):** "조립형 납품 모델(빼기·더하기 가능, 부품 자격 5조건을 모듈 분리의 완료 기준으로)을 정식 판매 방식으로 채택한다."
- **열린 질문(로드맵 §7)에 추가:** ① 고객 월렛 온보딩 경로(D4 — R1 VC 스파이크 결과가 ⓐ/ⓑ 기본값을 결정) ② 원화 레일 1급 승격(VASP 장벽 대응).
